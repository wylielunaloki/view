Getting Started with Git, Github and the Command Line Interface

What is "Git" and "Github"?

Git is a Version Control System used to track and manage programming and coding projects. It keeps a snapshot of all changes and updates made to a project (i.e. all versions of a project) so that changes can be tracked, edited and managed from any point in time. It also allows for others to collaborate on a shared project when it's uploaded to an online cloud-based platform like, Github.

Note: When developers are working on a project, that project is called a repository (or repo for short).

Github is a popular cloud-based platform that developers can use to store, edit and manage their repositories remotely. This is especially helpful when working on group projects or if developers need to use other computers (like a work computer) to access their projects. Turning a project into a "git repo" allows developers to upload local projects (i.e. projects made on their computer) onto a remote (online) platform.

The Command Line Interface or simply, the command line, is an way to interact with a computer program by writing text. The following commands are written in the command line.

Basic Git Commands:

Below is a list of git commands and a description of what they do.

Commands are shown here with `backtics` around them. Options use (-, --) dashes and give additional instruction for the command. Text found in between greater-than and less-than symbols (< ... >) signifies a blank space to add your own text in (often used when naming things).

`git init`
Turns a local project into a git repository so that it can be uploaded (or, pushed) online and edited remotely.
`git status`
Shows files that have been added or modified in a repository. Save changes in VSC by pressing "command+s". Any unsaved file in VSC will have a white dot that appears in the tab. Local files will need to be saved by typing esc and :w.
`git add`
Stages selected files (or all files if I simply type . instead of each individual file) to be committed. Don't forget to add a commit message -m "Adds a Commit message" -m "Adds additional content." If I do forget, the command line will ask me to add a message anyway. I need to write it in quotes for it to accept my message. Use single quotes for quoting in the command line when writing messages in double quotes! Duh!
`git commit`
Commits/makes those changes locally but doesn't publish those changes onto Github until I push them there. (For modified files, i.e. not newly created ones, I can skip `git add .` and use `git commit -am "commit message"` which adds and commits with a message at the same time.)
`git push`
Remember to save files locally before pushing them into a remote repo. (In VSC, type ctrl+s to save README.md, for instance. Or esc :w if in an html doc.) When pushing to an online repo for the first time, set an upstream location to push everything to. The command line will suggest how to establish an upstream location. Type `git push --set-upstream origin main` or `git push -u origin main` for short. Once that connection is established, easily publish all commits with `git push`. (However, you will need to connect local a repo with online repo first by using `git remote + ssh location.` See instructions in "git remote" below.)
`git pull`
The opposite of push. If I modify a file or add something to a repository on Github or another host, I can pull it from there and have those changes appear on my local machine. I do not need to add a location after the command like I would with git clone because the file in my local terminal should already be associated with the repo in Github. Assuming I have connected my local machine with Github via an ssh key.
`git clone`
If there is a repo that IS NOT on my local machine but I want to bring it down to use on my machine, I use the git clone plus location of the file (ssh clone address) which can be found in the "code" button on Github.
`git remote`
Allows a locally made repository to be uploaded to an online platform like, Github. First, create a repository on Github and name it. Then, in the command line, type `git remote add origin <ssh link or https link supplied by Github>` to connect it to the repository made online. Make sure changes are saved locally in VSC with "command+s", add and commit those changes. Finally, push those changes up to Github with the command, `git push --set-upstream origin main` which pushes all contents in the repository into the romote repo. The command, `git remote -v` shows any repositories that are connected to the local repo.
`git branch`
Let's me know what branch I am working in. To move to another existing branch, use `git checkout `.
`git checkout`
Let's me switch between branches within a repo. If I want to work on a side project, say in a README.md file within the repo, but do not want to push commits to the main project yet, I can create a new branch with `git checkout -b` and then name it with something like, <feature-README-instructions>. ("Git checkout" --> switches branches, "-b" --> an option used to create a new branch, "feature-" good way to start a branch name, "README-"--> the name of the file I am working on, "instructions-"--> additional description as I may have several branches named feature-README, or may be working in a team with others who are also working on README branches, for instance.) - Once the branch is created with -b, I can simply go back and forth between versions of README for instance in either branch (main or feature...) with `git checkout `. I can also stage, commit and push those changes onto Github without replacing the original main branch version of the side project until I am ready. - But before I can push commits from a new branch onto Github, I will need to establish an upstream location and tell it where to go. The command line will tell me how to do this, but it looks like this: `git push --set-upstream origin feature-README-instructions` or `git push -u ...` - I can create a pull request "PR", on Github when I am ready to merge branches. Then I need to use, `git pull` to bring that merge back down into my local main branch.
  • `git diff `
    • - Lets me know what things are different between branches. Useful before merging branches.
  • `git stash`
    • - Git will not let branches merge until changes between files are reconciled. Git stash let's you hold onto file changes without having to commit to those changes.
  • `git merge`
    • - Make sure to be in the branch that you want changes to merge into. The type, `git merge ` which merges the named branch into the branch you are logged into, in the terminal. * If master or main branch is being updated by others, you may not want to get too far behind those changes, therefore, try `git merge main` from withing the branch to make sure that before your PR is completed and branch is merged into main, you have the most updated version of main to merge into.
  • `git branch -d `
    • - This command deletes branches locally. Branches cannot be deleted from within them, so, make sure to checkout of the branch first. (And Make sure to pull any changes from a remote repo before deleting the branch.) The branch will need to be manually deleted on Github, too.
  • `git reset`
    • - To undo staged files (i.e. things added but not yet committed in git) type `git reset`. To undo committed files, type `git reset HEAD~1` where "HEAD" means the last commit, and `1` means one committ before the last commit. Therefore, to reset a file to three commits ago, use `git reset HEAD~2`.
  • `git log`
    • - Displays a log of all commits that have taken place. To go back to a particular commit and reset a file, type `git reset `. This unstages commits up to that point. To delete commits, type `git reset --hard ` which will delete all changes up to the commit number entered.
